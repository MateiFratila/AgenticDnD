- Python backend for an agentic D&D simulation. Core modules: backend/world, backend/agents, backend/llm, backend/orchestrator.
- World state is immutable and centralized in backend/world/state.py. Deterministic updates happen only through backend/world/dispatcher.py applying typed WorldMutation objects.
- Two-layer runtime design: Adjudicator (LLM decides legality/ruling + routing) -> Extractor (LLM maps approved ruling to mutations) -> Dispatcher (pure deterministic state updates).
- Orchestrator is the sole coordinator and sole caller of Dispatcher. Agents may propose; they do not mutate world state directly.
- TableOrchestrator runs a single-turn loop with steps: waiting_for_intent -> adjudicating -> extracting -> applying_mutations -> turn_complete.
- Phase 1 intent flow: TableOrchestrator now supports Intent -> Adjudicator -> Extractor. Empty `/api/advance` actions are generated by the intent agent, and active encounter NPCs can auto-resolve into `npc_turns` summaries behind hard-cap, repeated-state, and no-progress failsafes.
- Input normalization lesson: player free-text and intent-agent output now converge into `ResolvedAction` inside `backend/orchestrator/turn_models.py`, and `/api/advance` echoes the effective `action` used for adjudication.
- Encounter turn management now lives on `EncounterState` via `turn_order` + `current_turn_index`; when `active_encounter_id` is set, `TableOrchestrator` advances that encounter-owned pointer and auto-routes NPC slots through the Intent Agent until a PC is awaited again.
- NPC clarification handling now supports multi-pass self-resolution: if an NPC receives `needs_clarification` and the follow-up actor is still that NPC, the orchestrator may run additional intent passes for the same slot. Failsafes now use total internal pass caps, per-NPC follow-up caps, and repeated unresolved action detection.




- WorldState also carries session metadata: game_session_id (5-char id), active_actor_id, awaiting_input_from, world_version. This metadata is treated as part of the source of truth, not transient orchestrator-only state.
- Snapshot persistence is enabled by default in TableOrchestrator. Final world state is written after each loop to artifacts/world_snapshots/ using compact session-prefixed filenames: s_<game_session_id>_l_<n>_a_<actor>.json.
- AdventureLoader is snapshot-first. load_adventure() restores latest matching snapshot first; if none exists, it builds a fresh world from assets/ and assigns a new 5-char game_session_id.
- Fresh-world bootstrap detail: AdventureLoader now persists an initial snapshot immediately after building a new WorldState. This prevents generated game_session_id values from being lost before the first TableOrchestrator turn writes a snapshot.
- Loader restore nuance: _load_world_from_snapshot() still falls back to generating a new game_session_id when reading older/legacy snapshots that do not contain that field.

- Snapshot tools live in backend/orchestrator/snapshot_tools.py with CLI commands: list, diff, diff-latest.
- REST API wraps TableOrchestrator in FastAPI (backend/main.py creates app, backend/api/ contains routes + models).
- API lifespan initializes game engine on startup: loads world (snapshot-first), creates adjudicator/extractor agents, creates orchestrator.
- REST endpoint POST /api/advance: takes actor + action (freetext string: question or intent), routes through Adjudicator/Extractor, returns Outcome with status/ruling/turn_state.
- Not all responses advance turn (questions/clarifications stay on same actor; only approved actions advance).
- Orchestrator persists snapshots after each turn via api.
- README.md is the project operations guide. Rule: when a tool or command changes, update README in the same change set.
- Main regression scripts: python test_loader.py, python test_orchestrator.py, python test_snapshot_tools.py, python test_agent_contracts.py, python test_agents.py, python test_api.py.
- Practical testing note: tests that exercise loader/orchestrator/api should use isolated temp snapshot directories to avoid cross-test contamination because loader restores snapshots by default.
- Repo lesson: adjudicator prompts now use a scoped world view instead of a full world dump to keep token usage down while preserving decision-relevant context.
- Repo lesson: intent-agent payloads should use per-actor `actor_knowledge` stored on `WorldState`, so PCs/NPCs only see discovered rooms and observed creatures while the adjudicator can still receive a broader DM-scoped view.

- Repo lesson: `build_extractor_payload()` must resolve `active_actor_id` from both PCs and NPCs; otherwise NPC turns can produce `current_scene: null` and weaken extraction context.
- Repo lesson: extractor failures should flow bottom-up: when extractor returns explicit `[EXTRACTOR][needs_revision]` / extraction-failed feedback with no committable mutations, the orchestrator should send that feedback back through adjudication for a corrected or clarified ruling.
- Repo lesson: keep adjudicator payloads tidy by nesting helper fields under `world_state.summary` and the full truth under `world_state.canonical_world_state`, instead of duplicating many top-level views.


- Repo lesson: `turn_log` should preserve DM/adjudicator rulings for future-turn continuity; `[WORLD]` entries are still useful for state-change audit details.
- Repo lesson: LLM trace filenames must read session metadata from both top-level `world_state.*` and nested `world_state.session.*` fields to avoid `unknown` tokens.
- Repo lesson: LLM trace filenames should prefer `loop_index` over `turn_count` so repeated same-turn adjudication/extraction cycles do not overwrite prior traces.

- Repo lesson: unresolved rolls/checks should keep the same actor in `awaiting_input_from` and should not advance turn until the outcome is known.
- Repo lesson: unresolved-roll follow-up heuristics should inspect only the player-facing `ruling`, not technical `reasoning`; otherwise approved damage outcomes can be misclassified as `needs_clarification` and skip extractor commits.
- Repo lesson: adjudicator follow-up routing is now explicit via `AdjudicatorResponse.requires_player_response` and `follow_up_actor`; orchestrator routing should avoid prompt-text heuristics and trust structured contract fields plus `destination`.


- Repo lesson: room connectivity must be preserved in `WorldState.rooms[*].connections` and exposed in scoped payloads so navigation can resolve canonically (for example the single exit from `entrance` to `goblin_barracks`).
- Repo lesson: `/api/advance` uses the live in-memory orchestrator; rewinding persisted state requires rebuilding that orchestrator explicitly (now via `GET /api/rewind`, which deletes the newest session snapshot and restores the previous one).
- Repo lesson: dispatcher condition mutations (`add_condition` / `remove_condition`) must support NPC targets as well as PCs; otherwise extractor output can crash `/api/advance` with a `KeyError` on encounter enemies.
